# AICourse Codex 侧边栏划词解释计划书

日期：2026-06-29

分支：`codex`

## 1. 目标

在不破坏现有课程浏览体验的前提下，给 `aicourse` 增加一个 **Codex 侧边栏**，支持用户在课程页和 lesson 页中**划词提问、解释、翻译、举例、追问**。

这次改造的核心要求是：

- 前端 UI 和现有功能尽量不动，课程阅读主流程保持不变。
- 侧边栏以“辅助教师”的方式存在，不替代原有课程目录和正文。
- 默认中文优先，适配国内网络环境和中文学习场景。
- 交互要能从“选中文本”直接进入“问 Codex”。
- 解释结果要结合当前课程上下文，而不是只回答孤立片段。

## 2. 现状判断

当前仓库里，和这件事最相关的基础已经存在：

- `site/components/CourseShell.tsx` 已经有课程页面骨架。
- `site/components/Sidebar.tsx` 已经有课程目录侧栏。
- `site/app/[locale]/c/[repoId]/page.tsx` 和 `site/app/[locale]/c/[repoId]/lessons/[id]/page.tsx` 是课程首页和正文页。
- `site/components/Prose.tsx`、`site/components/StepSimulator.tsx`、`site/components/Mermaid.tsx` 是正文呈现的主要入口。
- `site/app/[locale]/j/[id]/page.tsx` 已经有任务页式的实时状态交互，说明站点里已有“边看边操作”的交互风格。

这意味着新功能不需要重做站点结构，重点是：

1. 找到一个不会破坏现有阅读布局的挂载点。
2. 把“划词”变成一个明确的交互入口。
3. 把上下文整理成 Codex 能稳定消费的输入。
4. 把回复结果以教师式中文表达出来。

## 3. 设计原则

### 3.1 不替换原有阅读体验

现有课程目录、正文、步骤模拟器、对比区块、参考资料都要保留。

Codex 侧边栏只能是“增量能力”，不能把课程页改成聊天页。

### 3.2 选区优先，手动提问兜底

主要入口是划词。

如果用户没选中文本，也可以通过“问这一节”“解释当前代码”“总结这一段”之类的快捷动作发起提问，但这是兜底，不是主路径。

### 3.3 中文优先，适合国内学习场景

默认输出中文，语言风格要符合“AI 教师”而不是“AI 助手闲聊”。

回答优先采用：

1. 先讲结论。
2. 再讲为什么。
3. 再讲例子。
4. 最后给下一步行动建议。

### 3.4 上下文必须可追踪

Codex 不应该只看用户选中的那一小段文本。

每次提问都要带上：

- 当前课程标题
- 当前 lesson 标题和编号
- 当前章节位置
- 选中内容
- 选区所在的正文块类型
- 必要时附带附近的代码片段或段落

### 3.5 低侵入实现

优先沿用现有页面结构：

- 桌面端用右侧抽屉或贴边面板。
- 移动端用底部抽屉。
- 选区提示用轻量浮层，不做大范围布局改造。

## 4. 功能拆解

### 4.1 必做功能

1. **划词触发**
   - 用户在 lesson 正文中选中一段文字后，页面出现轻量浮动动作条。
   - 动作条至少包含 `解释`、`翻译`、`举例`、`追问`。

2. **侧边栏打开**
   - 点击动作条后，Codex 侧边栏打开。
   - 侧边栏顶部固定显示选区摘要和上下文摘要。

3. **问题输入**
   - 选中内容自动进入输入框。
   - 用户可以直接修改问题，也可以追加追问。

4. **结果输出**
   - 结果以中文解释为主。
   - 输出结构建议固定成：`结论`、`解释`、`例子`、`你可以继续问什么`。

5. **上下文引用**
   - 解释内容要能引用当前 lesson 的标题、章节、代码文件名或当前步骤名。
   - 不是严格学术引用，但要让用户知道答案来自哪里。

6. **对话记忆**
   - 同一个 lesson 内的连续追问保留上下文。
   - 切换到别的 lesson 后，默认开启新会话。

### 4.2 建议功能

1. **快捷模式**
   - `解释这一段`
   - `翻译成白话`
   - `举一个例子`
   - `从初学者角度重讲一遍`
   - `出一道小题`

2. **内容感知**
   - 选中文本如果来自代码块，默认按代码解释。
   - 选中文本如果来自说明文字，默认按教学解释。
   - 选中文本如果来自图表说明，默认先讲图意再讲机制。

3. **答案分层**
   - 简短答案适合第一次打开。
   - 深入答案可以继续展开，但不要一上来长篇大论。

4. **收藏问题**
   - 允许把本次问答标记成“稍后复看”。
   - 方便学习时回到关键解释点。

### 4.3 非目标

这次不做成：

- 完整的全站聊天系统。
- 会话列表和复杂知识库产品。
- 自动修改课程内容。
- 自动重写 lesson。
- 自动生成新课程。

本次只聚焦“选中内容后，让 Codex 解释清楚”。

## 5. 页面落位

### 5.1 桌面端

建议保留现有左侧课程目录。

Codex 侧边栏放在页面右侧，作为可折叠抽屉存在，默认收起。

理由：

- 左侧已经承担目录导航。
- 正文区域是学习主内容。
- 右侧抽屉不会打断阅读路径。

### 5.2 移动端

移动端不强行做双栏。

建议使用底部抽屉或全屏半高面板：

- 选中文字后出现底部浮层。
- 点击后展开问答面板。
- 关闭后恢复原阅读状态。

### 5.3 课程首页与 lesson 页

lesson 页是主战场。

课程首页可以只提供一个入口按钮，不强制常驻对话区。

这样可以保持首页仍然是课程导览，而不是聊天主界面。

## 6. 交互流程

### 6.1 标准流程

1. 用户在 lesson 页选中文本。
2. 页面识别当前选区是否属于正文区域。
3. 右下角或选区附近出现动作条。
4. 用户点 `解释`。
5. 侧边栏打开，自动填入选区内容。
6. 用户确认提问方式后发送。
7. 侧边栏以流式方式展示回答。
8. 用户继续追问时，保持当前 lesson 的上下文。

### 6.2 代码块选区流程

如果选中的内容来自代码块，默认优先走：

- 这段代码做什么
- 每一行在干什么
- 为什么这样写
- 和课程目标的关系是什么

### 6.3 文本段落选区流程

如果选中的是说明性文字，默认优先走：

- 这段话在讲什么机制
- 这段话和前后文是什么关系
- 有没有更白话的说法
- 有没有一个小例子

### 6.4 空选区兜底

如果没有选区，也允许从侧边栏直接提问：

- 解释当前页面
- 总结这一节
- 讲一下这段代码
- 说出本节的难点

## 7. 数据与接口

### 7.1 前端输入

建议建立一个统一的上下文对象，至少包含：

- `courseId`
- `repoId`
- `locale`
- `lessonId`
- `lessonTitle`
- `sectionId`
- `sectionTitle`
- `selectionText`
- `selectionKind`
- `surroundingText`
- `codeFile`
- `codeLanguage`
- `activeStep`
- `conversationId`

### 7.2 后端请求

建议抽成一个独立的 assistant 接口，例如：

- `POST /api/codex/query`
- 可选 `POST /api/codex/stream`

请求体建议包含：

- 用户问题
- 选中文本
- 课程上下文
- 历史对话
- 输出模式

### 7.3 返回格式

建议返回统一结构，便于前端渲染：

```json
{
  "answer": "中文解释内容",
  "summary": "一句话总结",
  "highlights": ["关键点1", "关键点2"],
  "followUps": ["你还可以问什么"],
  "references": [
    { "label": "当前 lesson", "href": "/zh/c/..." }
  ]
}
```

如果使用流式输出，也要保证最终能收敛到同样的数据结构。

## 8. Codex 提示词策略

### 8.1 角色定义

Codex 在这里的角色不是普通聊天机器人，而是“课程教师”。

提示词要明确要求它：

- 用中文讲解。
- 不要只给定义，要讲原理和上下文。
- 不要编造仓库里没有的事实。
- 不确定时要明确说不确定。
- 优先围绕当前选区和当前 lesson 回答。

### 8.2 回答风格

推荐固定回答骨架：

1. `先说结论`
2. `再解释原因`
3. `再给一个例子`
4. `最后给下一步`

这更符合教学场景，也更适合国内用户的学习习惯。

### 8.3 国内可用性

模型接入层不要把前端和单一供应商绑死。

建议做成后端适配层，允许后续接：

- Codex
- OpenAI 兼容接口
- 国内可访问的模型代理
- 本地 fallback

这样可以保证：

- 线上环境可切换
- 国内网络环境可用
- 不因为某个 provider 不稳定就让功能完全失效

## 9. 前端实现建议

### 9.1 新增组件

建议新增：

- `CodexSidebar`
- `CodexTrigger`
- `SelectionBridge`
- `AssistantMessageList`
- `AssistantComposer`

### 9.2 状态管理

这类交互不需要一开始就上复杂全局状态库。

更稳妥的做法是：

- 页面级 React context 或轻量 reducer
- 按 lesson 隔离会话状态
- 选区状态和对话状态分开

### 9.3 选区获取

建议监听：

- `mouseup`
- `keyup`
- `selectionchange`

并对选区做限制：

- 必须在正文范围内
- 长度不能无限制增长
- 不允许跨越不相关 UI 区块

### 9.4 可访问性

侧边栏和动作条要满足基本可访问性：

- 键盘可达
- Esc 可关闭
- 焦点状态可见
- 屏幕阅读器能读到按钮语义

## 10. 风险与缓解

### 10.1 选区误触

风险：
- 用户拖选时频繁弹出动作条，影响阅读。

缓解：
- 只在有效选区稳定后再出现。
- 选区过短不触发。
- 在按钮交互区外点击自动收起。

### 10.2 侧边栏打断阅读

风险：
- 侧边栏太宽，压缩正文。

缓解：
- 默认收起。
- 只在需要时展开。
- 桌面端宽度固定，移动端改为抽屉。

### 10.3 回答失真

风险：
- Codex 根据选区发散，讲偏课程内容。

缓解：
- 提示词强制绑定当前 lesson 上下文。
- 回答中明确引用当前章节和代码位置。
- 必要时返回“我不确定，需要看更多上下文”。

### 10.4 国内网络可达性

风险：
- 某些模型或外部服务在国内环境下不可用。

缓解：
- 模型层抽象化。
- 后端支持配置不同 provider。
- 保证基础解释路径不依赖外部网页抓取。

## 11. 里程碑

### M1：计划与骨架确认

- 明确页面挂载点。
- 确定侧边栏是右侧抽屉，不替换左侧目录。
- 确认数据结构和接口草案。

### M2：选区与触发

- lesson 页支持划词。
- 出现动作条。
- 能把选中文本送进侧边栏。

### M3：问答面板

- 侧边栏可打开、可关闭、可输入。
- 能显示回答。
- 能保留一个 lesson 内的连续追问。

### M4：上下文增强

- 选区带上章节和代码上下文。
- 回答能引用当前 lesson 的具体位置。

### M5：流式与体验完善

- 支持流式输出。
- 支持移动端抽屉。
- 支持快捷提问和空选区兜底。

## 12. 验收标准

完成时至少要满足：

1. 用户能在 lesson 页选中文字并打开 Codex 侧边栏。
2. 用户能对选中内容发起解释请求。
3. 返回内容是中文，且能结合当前课程上下文。
4. 现有课程目录、正文、步骤模拟器等功能没有被破坏。
5. 桌面和移动端都能正常用。
6. 国内可用性方案已明确，不依赖单一外部服务。

## 13. 下一步建议

按这个顺序做最稳：

1. 先做页面挂载与状态设计。
2. 再做划词识别和动作条。
3. 再做侧边栏 UI。
4. 再接 Codex 查询接口。
5. 最后补流式、移动端和提示词优化。
